On Testing

I had the pleasure of being involved in a discussion on reddit — which is not something a person usually says. But the gist of it was this — comments are shortened, vote totals are approximate, and the names have been changed to protect the involved parties. To add context, this was a discussion on the topic of using unittest.mock to make testing easier.

/u/User1 [-1] “Don’t mock what you don’t own.”

/u/User2 [+5] “You don’t think mocking something like os.remove is a good move?”

/u/User1 [-6] “No, I think it’s bad architecture.”

/u/User2 [+7] “You’re insane then. :)”

This exchange has lead me to conclude two things:

  1. The phrase “unit test” is meaningless.
  2. Mocks are poorly understood and abused.

I offered this case for not mocking something like os.remove, with a bit of a text dump — something I’m often guilty of doing for better or worse.

Things like os.remove are rarely stand alone things, they’re often paired with stuff like “Format this file name.”, “Join this path.” “Can I find the file?”, “Do I have permission to remove the file.”

As well as these code samples:

from abc import ABC, abstractmethod
import os

class ReportRemover(ABC):
    @abstractmethod
    def remove_report(self, name):
        pass

class FileReportRemover(ReportRemover):
    def __init__(self, path, ext):
        self._path = path
        self._ext = ext

    def remove_report(self, name):
        filename = self._format(name)
        if self._is_removable(filename):
            os.remove(filename)
            return True
        return False

    def _format(self, name):
        filename = '{}.{}'.format(name, self._ext)
        return os.path.join(path, filename)

    def _is_removable(self, filename):
        return os.path.exists(filename) and os.path.isfile(filename)

And caveated that with I’d offer that rich class interface when I have multiple contexts to remove a “report” — maybe it’s generated from a database, maybe it’s on S3, maybe I’m running a test and don’t even have an actual report to remove. Conversely, I also offered this sample that I might use if I’m only going to have reports on the local filesystem:

from collections import namedtuple
Settings = namedtuple('Settings', ['DEFAULT_REPORT_PATH', 'DEFAULT_REPORT_EXT'])
SETTINGS = Settings('path/to/reports', 'html')


def remove_report(name, path=SETTINGS.DEFAULT_REPORT_PATH, ext=SETTINGS.DEFAULT_REPORT_EXT):
    filename = os.path.join(path, '{}.{}'.format(name, ext))
    if os.path.exists(filename) and os.path.isfile(filename):
        os.remove(filename)
        return True
    return False

Which was met with the question of, “Well how do you test that?” And this sort of astonished me. And maybe that’s because of my growth as a developer. But the answer seemed obvious to me, “Create a directory, put a file named test.report in it, run remove_report, assert that the directory is empty.”

import pytest
from things.reports import remove_report

def test_remove_report_deletes_file_report(tmpdir):
    report_dir = tmpdir.mkdir('delete_report')
    report = report_dir.join('test.report')
    report.write('') #needed to "create" the file

    remove_report(name='test', ext='report', path=report_dir.strpath)

    assert report_dir.listdir() == []

You can do the same thing with the class interface as well. Just import FileReportRemover and run FileReportRemove.remove_report with an instance of it instead. Maybe this person was under the impression that unit tests are the only type of test, or maybe the only valuable test. But what is the value of this test:

from unittest import mock

@mock.patch('os.path')
@mock.patch('os.remove')
def test_remove_report_deletes_file_report(osremove, ospath):
    ospath.join.return_value = 'path/test.report'
    ospath.isfile.return_value = True
    ospath.exists = True

    remove_report('test', 'report', 'path')

    assert ospath.join.call_args == (('path', 'test.report'),)
    assert ospath.isfile.called
    assert ospath.exists.called
    assert osremove.call_args == (('path/test.report'), )

There’s so much noise in this test. We’re setting up mocks, there’s strings all over, multiple asserts — which isn’t necessarily bad, but can be symptomatic of a poorly designed test. In fairness, there are three lines of setup in the other test, but I find that the setup is much clearer: We create a directory and put a file in it. In this one, we’re setting up mock return values. And we need to mock an entire submodule of os.path, otherwise the test will explode.

More over, it doesn’t get to the heart of the point of a test: are we actually removing a file from the file system. And it’s fragile, if the interface of os changes — unlikely, but that’s besides the point — the test will still pass. Using the autospec feature of mock will ease this pain, but I think that’s an anti-pattern in this case. We’re still creating a bunch of noise in the test.

And the worst part is this sets a precedence of mocking anything that’s inconvenient. All of the tests will look like this. Meaning, you’ll have to examine every piece of code under test to understand it. At that point, why even have tests? I’m being serious. I should be able to read the tests for some code and have a decent idea of what the implementation is.

The Incredibles, © Disney 2004

Of course, I got asked, “When would you mock?” And the answer to that is, “Any time I need to test that remove_report did something but I don’t actually need to remove a report.” I offered the concept of a report sweeper:

def test_report_sweeper_ignores_os_error():
    remover = mock.Mock(side_effect=OSError("You shouldn't see me"))
    reports = ['foo', 'bar', 'baz']

    report_sweeper(reports, remover=remover)

    assert remover.call_args_list == [mock.call('foo'), mock.call('bar'), mock.call('baz')]

Because I own the interface to report_remover, I can comfortably mock that dependency. An alternative approach avoids mocking altogether:

def test_report_sweeper_ignores_os_error_oddly():
    call_count = 0
    reports = ['foo', 'bar', 'baz']
    def remover(name, ext=None, path=None):
        nonlocal call_count
        call_count += 1
        raise OSError("You shouldn't see me")

    report_sweeper(reports, remover=remover)

    assert call_count == 3

I’ll be honest, I like the mock approach here more because it’s less noise. There is the mental overhead of knowing what Mock.side_effect does, but it’s such a useful thing to know that I’d rather someone look it up than understand the oddness that is the locally defined remover. But in either case, you can imagine what the implementation of report_sweeper looks like:

def report_sweeper(reports, remover=remove_report):
    for report in reports:
        try:
            remover(report)
        except OSError:
            pass

This is what a true unit test is. We want to assert that in the case of an OSError, report_sweeper ignores it. You don’t bother with fiddly details like, add a file to the system but give it the wrong permissions or just don’t add a file to your system, or whatever. Do we want an integration test? Maybe, but given an OSError is an OSError, maybe not.

Deeper Lessons

Going back to the original exchange that prompted this whole thing, I think there’s more important lessons to learn — or rather unlearn — about testing.

Unit Tests Aren’t The End All Be All

I love unit tests. I love being able to spin up a tiny portion of my application and make assertions about a laser focused behavior. “Given I want to remove some reports, when an OSError occures, then the sweeper ignores it.” That says nothing about what a report is, where they live or even how I’m removing it.

But on the other end, I need to be able to test that I can actually remove a report. And you can’t unit test this. It’s inherently tied to an external system — the file system, a database, the network. Maybe it’s tied to several of these external systems — we keep a list of reports in the database but they live in S3 buckets.

Indirection is Invaluable

Because I want to be able to unit test my report_sweeper, I don’t want to tie that to these external systems. Instead I want to tie it to a contract with my report remover: If you give me a name of a report, I’ll handle the orchestration of removing it and let you know if it succeeded.

The indirection is not directly calling into the os module, instead I call something that calls that for me. And this applies to all external collaborators. If you use requests to get some information, don’t call requests.get directly — have an adapter that manages all of that for you. I know this sounds crazy, but which is more descriptive read:

def get_some_data(id):
    data = requests.get('http://somedatasource.com/things/{}'.format(id))
    data.raise_for_status()
    serializer = SomeSerializer(data=data)
    serializer.is_valid(raise_exception=True)
    return serializer.data

Or:

def get_some_data(id):
    external = ExternalHelper(uri='https://somedatasource.com/things/{}', serializer=SomeSerializer)
    return external.fetch(id)

Even better is if get_some_data is actually a method on an object that lists ExternalHelper as a dependency.

Patching Is A Symptom

If you find yourself monkey patching stuff, step back and think, “Why?” Why are you patching that? It could be because you’ve tightly coupled your logic to requests (or any other external package). In that case, you should wrap up that communication. If you’re patching in that adapter, it could be because you’re testing a specific failure that’s hard to emulate — and generally, I’m okay with that. But it’s best to run that test as a live integration against that external system.

But if you find yourself patching parts of the object you’re actively testing, you need to stop and find out what you need to do to ease that relationship. By patching that method, you’re only going to bake in the badness and cause tremendous pain down the road.

Unit Test Logic

If there’s logic, test it. Test it every way you can think, and then get the nearest child to give it input and fix any holes they find. This will generally involve using stubs and fakes for expensive collaborators — things that do I/O, system calls and CPU warmers. But if you don’t need to fake something, don’t. Just use the real thing. It’s okay.

Boundary Test Relationships

This makes the assumption you wrap up external dependencies. When you’re testing relationships, you want to make sure Subject A talks to Subject B correctly, and vice versa. You’re testing things like, “Given I want Frob 314, Then I ask FrobRepository to fetch with 314 as the argument.” You want to test that the right arguments get to the right things and only add enough support code to get the test to finish.

Integration Test Wrappers

There’s generally very little logic in something that only serves to translate inputs and outputs from Domain A to Domain B. It’s usually things like this:

class Notifier(ABC):
    @abstractmethod
    def notify(self, recipient, message):
        pass

class TwilioTexter(Notifier):
    def __init__(self, sender, client):
        self.client = client
        self.sender = sender

    def notify(self, recipient, message):
        self.client.messages.create(to=recipient.phone_number, from_=self.sender, body=message)

You’ll recieve no value from all the mocking and patching in the world with this. You’ll derive some value from boundary testing this by supplying a fake client that records its inputs. But the most value will come from actually sending a text message with a test.

Now, you’ll not want to always send a text message when you run your test suite, especially if you’re not poking something related to this wrapper. Luckily, py.test provides a very helpful marker module and a way to specify when to run these markers:

# tests/.../test_twilio_wrapper.py

@pytest.mark.costs_money
@pytest.mark.integration
class TestTwilioWrapperSuite(object):
    pass

And on the command line, you can specify like this:

py.test -m "not integration or system"

to only run unit tests, or

py.test -m "integration and not costs_money"

To only run integrations that aren’t also tagged with the costs_money tag. Or you could only tag it with costs_money to avoid accidentally running it with your integration tests. Another big helper here is tox, which goes a long way to setting up and executing tests for you. I love it so much, I’ve bound it to <F12> in normal mode for Vim. Alternatively, you can simply run !tox -e comma,separated,envs as well.

This allows you to run fast moving unit tests all the time and slow moving integration tests and system tests sometimes.

System Tests Are For The Big Picture

You don’t want to do fine grained testing in your system tests. These are tests that integrate with things like frameworks, threads, asyncio, etc. And require specialized setup and the whole production application (though not necessarily the production data). These would be things like, “When I send a notification throught Twilio, the SMS sink shows the message as recieved”.
