Getting start with aiohttp.web: A todo tutorial

aiohttp is an HTTP toolkit based on the Python 3.4+ asyncio module. It includes:

  • A requests-like HTTP client
  • A high level HTTP server interface meant for building web applications
  • A low level HTTP server for web servers that don't need the conveniences of the high level server

The difference between the high and low level servers is that the high level server offers things such as:

  • Routing
  • Signals for startup, loop available and shutdown
  • Middlewares
  • Mounting subapplications (similar but not quite the same as Flask's blueprints)

The low level server still allows the use of the high level request, response and websocket objects so you're not so low that you'll dealing with TCP sockets directly.

In this guide, we'll build a simple Todo application as a short introduction to getting started with the high level server. In the future, I hope to cover more complex applications, but the todo list has become the fizzbuzz of web programming.

Installation

You'll need at least Python 3.6. aiohttp does run on Python 3.4, but I much prefer the async and await functionality introduced in Python 3.5. The reason for using 3.6 is making using the async comprehensions introduced in that version. Other than that, everything here should run in 3.5 (and 3.7 if you're into alpha builds).

Once you've ensured that at least 3.6 is installed, you need to install aiohttp. I suggest making a virtualenv for this:

python3.6 -m venv asynctodo
source asynctodo/bin/activate
python -m pip install aiohttp==2.0.7  # current version as 2017-04-16

The basic application

We're going to start small with a simple in memory list of dictionaries and routes that will display it and individual items from it. Since this is an inmemory object and we'll be restarting the server a few times (aiohttp does not come with a reloader but I'll cover that later). I am naming this file aiotodo.py and will reference this file name several times, if you decide to name it something else, just subsitute that name in where ever aiotodo.py or aiotodo appears.

from aiohttp import web

TODOS = [
    {
        'name': 'Start this tutorial',
        'finished': True
    },
    {
        'name': 'Finish this tutorial',
        'finished': False
    }
]

def get_all_todos(request):
    return web.json_response([
        {'id': idx, **todo} for idx, todo in enumerate(TODOS)
    ])

def get_one_todo(request):
    id = int(request.match_info['id'])

    if id >= len(TODOS):
        return web.json_response({'error': 'Todo not found'}, status=404)

    return web.json_response({'id': id, **TODOS[id]})

def app_factory(args=()):
    app = web.Application()
    app.router.add_get('/todos/', get_all_todos, name='all_todos')
    app.router.add_get('/todos/{id:\d+}', get_one_todo, name='one_todo')

    return app

There's a little bit going on here. Let's break it down.

aiohttp.web is where all the high level application server components live. It provides things such as the application class, the request and response classes, and helpers like json_response which is a shortcut to creating application/json responses without needing to remember to manually dump the object graph yourself.

get_all_todos and get_one_todo are our route handlers. You'll notice they aren't async -- this is alright as the server is capable of dealing with both sync and async handlers. Handlers should accept the request in a Django style. However unlike Django and Flask (but apparently similar to Pyramid), url parameters are accessed through the request.match_info mapping. The request object contains everything relevant about a particular request including a reference to the application that produced it.

Update /u/nerdwaller pointed out that route handler should always be async to encourage you to think in coroutines. I've left the original code here to be coherent with the text, however, to transform them into coroutines all that would change is using async def rather than just def. I have to agree that this promotes a nice symmetry to the other route handlers even in the simple case of dumping a list to json.

The {'id': id, **todo} syntax, if you're unfamiliar, was added in 3.5 and allows a mapping to be unpacked into a dictionary literal. The same change was made for sequences and list, set and tuple literals as well (using * unpacking as opposed to ** unpacking). This is very useful extending copies of dictionaries.

app_factory is more than a nicety for grouping all the application construction together, it's need for both invoking the aiohttp dev server and for the gunicorn aiohttp workers. You'll notice it accepts args, when executed from the command line, arguments that the aiohttp entrypoint doesn't understand are passed here -- we'll be using that later.

Finally, application routing is something of a mix of Flask and Django. It uses {name:pattern} to capture URL parameters. The name portion becomes available in the request.match_info dictionary, whereas the pattern portion is a regex. However, while it's vaguely similar to the Flask (actually Werkzeug) routing pattern, there is no notion of a converter so conversion must done manually in the route.

Cavaets

There are a few things I want to warn about here:

  • Even though the aiohttp.Application can use synchronous route handlers, you should take care to never do anything that would block the event loop powering the application. This includes synchronous IO and CPU intensive tasks.
  • With the pattern portion of URL parameters, any unmatched { or } will cause the router to refuse building the route with an exception. However, using matched {} pairs (e.g. \d{2}) will work. See Issue 1778 on aiohttp for some more details.
  • aiohttp's default router does not enforce any sort of automatic trailing slash redirection, so /todos and /todos/ are considered separate endpoints. Be careful with this if you are expecting it to handle trailing vs no trailing / for you
  • Finally, the note about regex matching in routes above only applies to the default aiohttp router. There are other implementations of it (in fact, the maintainers of the library encourage you to try them).

Running the application

Now that we have a basic application, we have something we can run with the aiohttp devserver. In the same directory as aiotodo.py and with your virtualenv active (source asynctodo/bin/activate) run:

python -m aiohttp.web -P 8080 aiotodo:app_factory

You should see:

======== Running on http://localhost:8080 ========
(Press CTRL+C to quit)

When you do, you can open a browser and navigate to localhost:8080/todos/ (note the trailing /) and see the two initial items we put in the list along with their ids. You can also navigate to localhost:8080/todos/0 and localhost:8080/todos/1 (note no trailing /) to view individual todos.

Creating, Updating and Deleting Todos

To actually be able to interact with the Todo List, we need to add a few more handlers:

async def create_todo(request):
    data = await request.json()

    if not 'name' in data:
        return web.json_response({'error': '"name" is a required field'})

    if not isinstance(name, str) or not len(name):
        return web.json_response({'error': '"name" must be a string with at least one character'})

    data['finished'] = bool(data.get('finished', False))
    TODOS.append(data)
    new_id = len(TODOS) - 1

    return web.Response(
        headers={'Location': str(request.app.router['one_todo'].url_for(id=new_id))},
        status=303
    )

Most of this is (hopefully) straight forward. We need to await the json parsing as it is an IO operation. We also return a response with no body but a redirect status of 303 SEE OTHER and the Location header set to where the client should go to. The request.app.router[...].url_for() is how you construct URLs to other endpoints with the aiohttp server -- the lookup on the router corresponds to the name='...' argument to the route constructor (add_get etc).

There is an assumption here is that clients will automatically follow the redirect to where we created the todo at. If not (curl for example), they will need to read the Location header and manually handle it.

Updating and deleting are similar:

async def update_todo(request):
    id = int(request.match_info['id'])

    if id >= len(TODOS):
        return web.json_response({'error': 'Todo not found'}, status=404)

    data = await response.json()

    if 'finished' not in data:
        return web.json_response({'error': '"finished" is a required key'}, status=400)

    TODO[id]['finished'] = bool(data['finished'])

    return web.Response(status=204)

def remove_todo(request):
    id = int(request.match_info['id'])

    if id >= len(TODOS):
        return web.json_response({'error': 'Todo not found'})

    del TODOS[id]

    return web.Response(status=204)

Hopefully there's nothing surprising here. Just deleting the item out of the list is a little questionable -- we could replace it with None and in both of the get handlers deal with that.

The biggest difference between the create and these endpoints is that the create returns a redirect to where the todo ended up and these return a No Content status. The reasoning being that a client won't know where the new todo is so we should tell them (they could consult the main list, but why force them), whereas since the client just acted upon the individual todo, they know where it is -- or "was" for the delete, in this implementation it's simply been replaced by the next or is now out of bounds.

Now, to hook up these handlers to the application:

def app_factory(args=()):
    app = web.Application()
    app.router.add_get('/todos/', get_all_todos, name='all_todos')
    app.router.add_post('/todos/', create_todo, name='create_todo')
    app.router.add_get('/todos/{id:\d+}', get_one_todo, name='one_todo')
    app.router.add_patch('/todos/{id:\d+}', update_todo, name='update_todo')
    app.router.add_delete('/todos/{id:\d+}', remove_todo, name='remove_todo')
    return app

Now run the server again (stopping the old one if you didn't already) and use your favorite tool to make POST, PATCH and DELETE requests against these new endpoints. You now have a fully functional todo list application -- well, for some clients, I'll leave the Web 2.0 javascript client as an exercise for you to implement or a potential future post.

Making it persistent

I really don't like blogs that take you halfway to a solution and then leave you with obvious questions like, "What if I wanted to make this persistent?" So let's go there.

The next thing we'll need to do is install some postgres related packages. The first is psycopg2, I've linked the install documentation in case a simple pip install psycopg2 does not work. In addition, we'll want to install aiopg and sqlalchemy.

Make sure the virtualenv is activated:

python -m pip install psycopg2 aiopg sqlalchemy

The next thing you might wonder is why aiopg over asyncpg if asyncpg is roughly 3x times faster? Application speed isn't everything, I value my time for writing and maintaining code over the run time of an application. On top of that, writing raw SQL can be tedious. If I could determine that aiopg was in fact a bottleneck and asyncpg would fix it, that when I would determine that change.

And honestly, if I was truly valuing speed of the application, I would not be writing it in Python in the first place.

Running a database

I've chosen Postgres here because it's a personal favorite. There is also aiomysql. And bindings for SQLite as well. Either way, getting access to a database is up to the reader. I run Xubuntu so I ended up using the Postgres docker image:

mkdir -p tmp/pgdata
docker run -d --name postgres -p 5432:5432 \
    -v $(pwd)/tmp/pgdata:/var/lib/postgres/data \
    -e POSTGRES_USER=aiotodo -e POSTGRES_PASSWORD=12345 -e POSTGRES_DB=aiotodo \
    postgres

If you're confused by that, it's okay. However explaining docker, volumes and other things about this are out of scope for this article. The short of it is Linux magic fairy dust and then a postgres database appears.

Connecting the application to the database

With the packages installed and the database running, we can start connecting to it from the application. The first thing will be writing the code that will make the table we want to use. We'll also create the SQLAlchemy view of the table as well.

from aiopg.sa import create_engine
import sqlalchemy as sa

metadata = sa.MetaData()
todo_tbl = sa.Table(
    'todos', metadata,
    sa.Column('id', sa.Integer, primary_key=True),
    sa.Column('name', sa.String(255), unique=True, nullable=False),
    sa.Column('finished', sa.Boolean(), default=False, nullable=False)
)


async def create_table(engine):
    async with engine.acquire() as conn:
        await conn.execute('DROP TABLE IF EXISTS todos')
        await conn.execute('''CREATE TABLE todos (
            id SERIAL PRIMARY KEY,
            name VARCHAR(255) NOT NULL UNIQUE,
            finished BOOLEAN NOT NOT NULL DEFAULT FALSE
        )''')

When's the last time you wrote DDL by hand? There are ways to hook alembic up to this application and have that manage creating tables for us, but that's something I'll cover another time.

If you've only ever been exposed to SQLAlchemy's ORM layer, you've maybe written something like this before for many-to-many relationships, but there's an entire layer of SQLAlchemy that stradles the area between raw SQL and ORM called the Core layer. And that's what we'll be using for this section. Now for things that make the application interact with the database:

async def attach_db(app):
    app['db'] = await create_engine(
        ' '.join([
            # this comes from the docker run command above
            # yours may vary depending on the database you're using
            # and how you configured the user, password, and database name
            'host=localhost',
            'port=5432',
            'dbname=aiotodo',
            'username=aiotodo',
            'password=12345'
        ])
    )

async def teardown_db(app):
    app['db'].close()
    await app['db'].wait_closed()
    app['db'] = None


async def populate_initial_values(engine):
    async with engine.acquire() as conn:
        await conn.execute(todos_tbl.insert().values({'name': 'Start this tutorial', 'finished': True}))
        await conn.execute(todos_tbl.insert().values({'name': 'Finish this tutorial', 'finished': False}))

async def setup_todo_table(app):
    await create_table(app['db'])
    await populate_initial_values(app['db'])

All of these functions are going to be signal handlers in the application there are four default signals that we can integrate with. All of them recieve the application as its sole argument.

  • on_loop_available: fired when the loop becomes available in a synchronous manner, so any async work needs to use the application loop explicitly. This is the only (current) handler that is synchronous.
  • on_startup: fired right before the application begins, this is very useful for setting up background tasks (such as long polling tasks)
  • on_teardown: fired after the application has recieved a shutdown signal from the caller and requests have finished. Here we should teardown anything we have setup and close long running connections to remote services.
  • on_cleanup: fired after teardown has finished and allows final cleanup steps to run, for example tearing down objects that we couldn't in the teardown setup because of dependencies between components

I find that on_startup and on_teardown are the ones I use most, however I have needed to use on_loop_available once or twice (for example, an application that integrates zeep into aiohttp as its start up uses the loop to do work rather than await).

To hook these handlers up to the application, we simply append them to the signal we want to use. Additionally, we'll add an optional argument to the application factory that determines if the table creation and initial population of it should run.

def app_factory(args=()):
    app = web.Application()

    app.on_startup.append(attach_db)
    app.on_teardown.append(teardown_db)

    if '--make-table' in a:
        app.on_startup.append(setup_todo_table)

    app.router.add_get('/todos/', get_all_todos, name='all_todos')
    app.router.add_post('/todos/', create_todo, name='create_todo')
    app.router.add_get('/todos/{id:\d+}', get_one_todo, name='one_todo')
    app.router.add_patch('/todos/{id:\d+}', update_todo, name='update_todo')
    app.router.add_delete('/todos/{id:\d+}', remove_todo, name='remove_todo')

Now let's re-run the application with the new command to create the table:

python -m aiohttp.web -P 8080 --make-table aiotodo:app_factory

Assuming we've done everything correctly the application will start up just fine. All of the data is coming from in memory list for now, but we've successfully hooked the application up to the database. When you cancel out of the server you should see no warnings or exceptions as well.

I'll note that --make-table will always drop the existing todo table when used. If you're making changes to the code and want to preserve the data be sure to remove that flag.

Manipulating the database from handlers

Retrieving, updating, inserting and deleting the Todos from the database is fairly straightforward. Since the request we receive in the handler has access to the application which has a connection pool on it, it's mostly of adding some async with blocks in place of list manipulations. Not all of this is super performant or what you'd want to do for a production application but for a first pass it's not that bad.

First we'll do the easy ones, fetching data from the database:

async def get_all_todos(request):
    async with request.app['db'].acquire() as conn:
        todos = [dict(row.items()) async for row in conn.execute(todo_tbl.select().order_by(todo_tbl.c.id))]
        return web.json_response(todos)

async def get_one_todo(request):
    id = int(request.match_info['id'])
    async with request.app['db'].acquire() as conn:
        result = await conn.execute(todo_tbl.select().where(todo_tbl.c.id == id))
        row = await result.fetchone()

    if not row:
        return web.json_response({'error': 'Todo not found'}, status=404)

    return web.json_response(dict(row.items()))

These should be drop in replacements for the original versions. The biggest change that might be not be apparent is in get_one_todo. First we fetch a pointer to a result set then we await the first result from that set. Finally, we need to check if we actually got a result or nothing. We could iterate over the result, as in get_all_todos but using fetchone makes it apparent we're expecting exactly one result.

If you run the application now, you should get the same output for the two get calls.

We've seen async with a few times and now async for has been introduced so I should take some time to explain those.

async with

If you're familiar with context managers, e.g. the with statement, async with is an asynchronous context manager. The biggest difference is that instead of def __enter__ and def __exit__ we use async def __aenter__ and async def __aexit__. This means that we can await things in the setup and teardown phases of the block. Here it's being used to get a database connection and later terminate it.

You may have also seen constructs such as with await something (or with (yield from something) in 3.4). This is completely different as this is a coroutine that returns a context manager.

async for

Similar to async with, async for is the asynchronous version of iteration. This allows us to iterate over something that needs to do async IO to produce values. There have been some churn in the async iteration protocol, but as of today (2017-04-16) the interface is def __aiter__ returning an object that has async def __anext__ and def __aiter__. Previously, it was async def __aiter__ returning the same sort of object. Confusingly, this change was made in 3.5.2, so you may see warnings about an async for construct returning an awaitable rather than an async iterator.

In this code, we're making use of async comprehensions which allows an asynchronous iterator to power comprehensions directly rather than needing to do:

results = []
async for x in things:
    results.append(x)

The restriction being that you can only use them inside of a coroutine. This change also supports awaiting coroutines in a comprehension as well and async generator expressions (if you want super lazy evaluation of values).

I digress though. Let's get back to updating the other end points.

Updating and Deleting

These are also straightforward with just replacing the original list manipulation with database connections:

async def remove_todo(request):
    id = int(request.match_info['id'])

    async with request.app['db'].acquire() as conn:
        result = await conn.execute(todo_tbl.delete().where(todo_tbl.c.id == id))

    if not result.rowcount:
        return web.json_response({'error': 'Todo not found'}, status=404)

    return web.Response(status=204)

async def update_todo(request):
    id = int(request.match_info['id'])
    data = await request.json()

    if 'finished' not in data:
        return web.json_response({'error': '"finished" is a required key'}, status=400)

    async with request.app['db'].acquire() as conn:
        result = await conn.execute(
            todo_tbl.update().where(todo_tbl.c.id == id).values({
                'finished': bool(data['finished'])
            })
        )

    if result.rowcount == 0:
        return web.json_response({'error': 'Todo not found'}, status=404)

    return web.Response(status=204)

These are mostly the same except instead of checking the ID before doing other checks, we reverse it:

  • Check that we have everything we need
  • Manipulate the database
  • Check the rowcount attribute to see if we actually did anything

An important note about rowcount is that it returns how many rows matched the query, not how many were changed by it. For a delete, this will always be the same, but for updates there's no guarantee for that.

I suggest running the application again and playing with these two new endpoints (remember --make-table flag deletes the existing data).

Inserts

This one is a little more complicated that the others as aiopg.sa doesn't implement everything that the SQLAlchemy core layer does.

from sqlalchemy import sql


async def create_todo(request):
    data = await request.json()

    if 'name' not in data:
        return web.json_response({'error': '"name" is a required field'})

    name = data['name']

    if not name or not isinstance(name, str):
        return web.json_response({'error': '"name" must be a string with at least one character'})

    todo = {'name': name, 'finished': bool(data.get('finished', False))}

    async with request.app['db'].acquire() as conn:
        async with conn.begin():
            await conn.execute(todo_tbl.insert().values(todo))
            result = await conn.execute(
                sql.select([sql.func.max(todo_tbl.c.id).label('id')])
            )
            new_id = await result.fetchone()

    return web.Response(
        status=303,
        headers={
            'Location': str(request.app.router['one_todo'].url_for(id=new_id.id))
        }
    )

The reason this is more complicated than what would be in a typical SQLAlchemy Core insert and return the new id is that aiopg.sa doesn't implement the inserted_primary_key attribute that the SQLAlchemy Core does, instead we need to go back to the database ourselves and fetch the largest id from the table. We do all this inside of a transaction to prevent a race where two people submit a new todo at the same time and both end up being redirected to the same todo item.

Since I'm unfamiliar with the finer points of the Core layer, I couldn't make the documentation example work:

await conn.execute(tbl.select(sql.func.max(tbl.c.id).label('id')))

This resulted in an exception about how the expression wasn't tied to a connection and after tinkering a bit, that changed to an exception about how aggregate functions cannot be used in a WHERE clause, even though I'm not 100% why SQLA was trying to do that.

For those not in know, an aggregate function is any SQL function that rolls up many values into one. They can't be used in a WHERE clause, but they can be used in a HAVING clause (which is basically WHERE for aggregate values).

The takeaway

Hopefully this will encourage you to start writing some web services using the asyncio and surrounding ecosystem. I've been using the aiohttp webserver for the last few months in various applications. I will admit to missing some things that are just there in the Flask and Django ecosystems, but there's nothing that can't be adapted or rebuilt to fit into the aiohttp world.

There's quite a few things I haven't covered here that could be used to improve and/or extend the functionality of this code:

  • Class base views for organizing the routes that have multiple HTTP methods attached to them
  • Websockets could be useful for synchronizing state across multiple clients. For example, you add an item and I recieve an update about it.
  • Using aiohttp-devtools to get autoreloading of the application's code
  • Or testing -- pytest-asyncio is indispensable for this, and aiohttp ships with testing utilities for its server (though oddly not for the client side stuff)
  • Mistakes when dockerizing Python apps
