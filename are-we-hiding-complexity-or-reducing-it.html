Are we hiding complexity or reducing it?

Consider this seemingly innocuous Flask endpoint.

from myapp.token import TokenGenerator
from myapp.decorators import email

@app.route('/send-reset-password-token/<username>')
@email
def generate_auth_token_and_send_it(username):
    user = User.query.filter_by(username=username).first_or_404()
    token = TokenGenerator(secret_key=current_app.secret_key).dumps({'op': 'reset-token', 'username':user.username})
    html_body = render_template('token-reset-email.html', token=token, user=user)
    return Message(body=html_body, receiver=user.email)

And, let’s pretend the email decorator looks like this:

from myapp.extensions import emailer

def email(f):
    def decorator(*args, **kwargs):
        message = f(*args, **kwargs)
        emailer.send(message, sender=current_app.config['ADMIN_EMAIL'])
        return redirect('front.homepage')
    return decorator

We might look at that and think, “Hey! That’s pretty cool, sending the email is separated from composing the email! It’s testable!”

Is it though?

What does the email decorator actually do? It intercepts a request from some endpoint and takes the message it returns and sends it. Then it redirects to some homepage. It’s still coupled, we can’t test either part without cranking up Flask and running the test server and worse still the logic is separated oddly.

The creation of the message is the endpoint’s responsibility. But the sending of the email resides some place else. The end point obviously wants to send a notification to a user, but it doesn’t. Instead it ends up just going “Well….uh…here’s this message thing. I hope whoever gets it knows what to do with it.” Similarly, the email decorator wants a message but it doesn’t build one, it gets one from someplace else.

Mostly this tell us that the logic inside the email decorator was originally inside this endpoint, but someone got tired of typing emailer.send(msg, sender) all the time and decided that was the seam to split on. I’m not gonna cast stones, I’ve written things like this. My eyes would glaze over and became limp every time I’d need to do emailer.send(msg, sender).

However, all this decorator has accomplished is hiding the complexity instead of reducing it. Remember when you were young and your mom or dad would tell you to clean your room. And instead of cleaning it, you shoved everything into the closet and under the bed? Your room looked clean, but you only hid the mess.

…maybe you didn’t do that, but I remember doing that. And I do the same thing in my code. But now, instead of shoveling the complexity into decorators and pushing it away and never dealing with it, I want to look at the complexity and think, “What’s the best way to clean this and keep it from happening again?”

How can we fix it?

Before we touch anything, we should create a test harness (something I’m trying to force myself to do lately).

from unittest import mock
import pytest
from myapp import create_app
from myapp.auth.views import generate_auth_token_and_send_it

@pytest.fixture(scope="session")
def app():
    return create_app(config='test')


@pytest.yield_fixture
def app_context(app):
    with app.app_context():
        yield app

These two things are a staple in my conftest.py for Flask applications. The first creates a single Flask app for the entire session. This might not be the best way to handle testing Flask apps, but it certainly speeds things up. If you don’t have environment senstive things (checking for Redis, for example), this probably good enough for end-to-end testing, if you have that sort of thing…well, you’ll probably want to create things create_app_with_redis or maybe use a parametrized fixture. I’m not going to get into that here though.

The app_context fixture is local to each function that wants it. It’s much quicker to set up and teardown application contexts than a whole application, which makes sense.

Next, we’ll want to batter our endpoint with tests. This part is no fun because we need to essentially test the endpoint through the UI.

from myapp.models import User, db
from inspect import unwrap

def test_generate_auth_token_and_send_it_calls_emailer(app_context):
    fred = User(username='fred', email='fred@fred.fred')
    db.session.add(fred)
    db.session.commit()

    with mock.patch('myapp.decorators.emailer.send') as sender:
        generate_auth_token_and_send_it('fred')

    assert sender.called

def test_expected_response_from_generate_auth_token_and_send_it(app_context):
    actual_function = unwrap(generate_auth_token_and_send_it)

    fred = User(username='fred', email='fred@fred.fred')
    db.session.add(fred)
    db.session.commit()

    message = actual_function('fred')

    assert 'Your user token is' in message.body
    assert fred.email == message.receiver


def test_failure_with_nonexistent_user(app_context):
    actual_function = unwrap(generate_auth_token_and_send_it)

    # first_or_404 calls abort,
    # intercepting that is easier than mocking the query
    with mock.patch('flask_sqlalchemy.abort') as aborter:
        actual_function('no freds')

    assert aborter.called

The very first thing you should notice is that these tests are hard to write. There’s all sorts of setup, things are being mocked left and right. We have to mock something in the Flask-SQLAlchemy library! If nothing else sends up red flags, mocking something in a third party library should.

However, there’s something important we’ve just learned: our design is bad. Trying to isolate this one function, and let’s be honest there’s things we’ve missed here, has put tremendous pressure on our thinking. The only reason I knew first_or_404 calls flask.abort is because I’ve contributed to the library.

There’s also another, more insidious dependency we have: Python 3.4+

  • Python 3.2 added the __wrapped__ attribute for functions wrapped with update_wrapper/wraps
  • mock was added to unittest in 3.3. It is available as a third party library though.
  • inspected.unwrap was added in 3.4, which can unwrap decorators using __wrapped__

So these tests can’t run on almost every version of Python. We could ditch unwrap and simply call dig through to __wrapped__ ourselves, but there could be more decorators between @email and the actual function that get added in the future, breaking the tests and they’ll be fixed by just doing this function.__wrapped__.__wrapped__.__wrapped__. The other option is implementing unwrap ourselves, which brings our dependencies down to just Python 3.2

How to fix it.

The first step is to stop hiding the complexity. And to me, that means bringing the emailer back into the endpoint.

@app.route('/send-reset-password-token/<username>')
def generate_auth_token_and_send_it(username):
    user = User.query.filter_by(username=username).first_or_404()
    token = TokenGenerator(secret_key=current_app.secret_key).dumps({'op': 'reset-token', 'username':user.username})
    html_body = render_template('token-reset-email.html', token=token, user=user)
    msg = Message(body=html_body, receiver=user.email)
    emailer.send(msg, sender=current_app.config['ADMIN_EMAIL'])
    return redirect('index.homepage')

The reason for this is that msg = Message(...) and emailer.send(...) form a logical unit. They want to be near to each other, originally they were in a long distance relationship and we all know long distance relationships are hard. We could simply pull both the message construction and the email sender out into a decorator, but then the function name is wrong. It’s no longer generate_auth_token_and_send_it it’s generate_auth_token_and_return_some_structure_a_decorator_deals_with.

By moving that logic back inside the endpoint, we’ve eliminated our hard dependency on Python 3 (there’s nothing to unwrap right now, and mock can be pip installed). Things are looking up already. If you’re modifying your tests along with your source code, you can remove the unwrap lines now.

What’s the use case?

The next step is to look at the story the endpoint is telling us:

  • Given a username, the endpoint locates the proper user
  • When the user is found, the endpoint generates a token for password resets for the user
  • Then the endpoint generates an email
  • The endpoint sends the email
  • Finally the endpoint redirects to the homepage

There’s also an error case:

  • If the endpoint can’t find the user, it redirects to an error page

By doing this, we can completely pull the actual logic out into it’s own function:

def send_user_reset_token(user):
    token = TokenGenerator(secret_key=current_app.secret_key).dumps({'op': 'reset-token', 'username':user.username})
    html_body = render_template('token-reset-email.html', token=token, user=user)
    msg = Message(body=html_body, receiver=user.email)
    return emailer.send(msg, sender=current_app.config['ADMIN_EMAIL'])

@app.route('/send-reset-password-token/<username>')
def generate_auth_token_and_send_it(username):
    user = User.query.filter_by(username=username).first_or_404()
    send_user_reset_token(user)
    return redirect('index.homepage')

That’s even better. Our test through the UI can be replaced with something a little simpler:

def test_do_we_send_the_token(app_context):
    fred = User(username='fred', email='fred@fred.fred')
    db.session.add(fred)
    db.session.commit()

    with mock.patch('myapp.auth.view.send_user_reset_token') as sender:
        generate_auth_token_and_send_it('fred')

    assert sender.called

I’m not completely happy with this test, but it’s okay. However, before we completely divert our attention from the endpoint test, let’s fix the second test where we mock flask_sqlalchemy.abort:

@app.route('/send-reset-password-token/<username>')
def generate_auth_token_and_send_it(username):
    user = User.query.filter_by(username=username).first()

    if not user:
        flash('Username not found')
        return redirect('generate_auth_token_and_send_it')

    send_user_reset_token(user)
    return redirect('index.homepage')

# in the tests
from flask import get_flashed_messages

def test_failure_with_nonexistent_user(app_context):
    with mock.patch('myapp.auth.view.send_user_reset_token') as sender:
        generate_auth_token_and_send_it('fred')

    assert not aborter.called
    assert 'Username not found' in get_flashed_messages()

We still need to spin up Flask to run these tests, but they’re simpler. The negative assertion in this test can be dangerous, but it’s paired with a positive assertion so I feel fine with having it.

That leaves just the emailer and the token generator to deal with. The reason for emailer to be mocked out is obvious, but the token generator might be less obvious. If it’s a good token generator, you won’t be able to know ahead of time what it’ll generate, it also has a hard dependency on our application. Both of these make it hard to test.

The emailer problem I’ve addressed else where, we just build an interface we want to use and pass the implementation in. We’ll do the same for the token generator. The dumps/loads interface is pretty common, so we could even subsitute the regular json module for it. So the actual logic really becomes:

def send_user_reset_token(user, tokenizer, notifier):
    token = tokenizer.dumps({'op': 'reset-token', 'username': user.username})
    payload = 'Your reset token is {0}'.format(token)
    return notifier.notify(user, token)

@app.route('/send-reset-password-token/<username>')
def generate_auth_token_and_send_it(username):
    user = User.query.filter_by(username=username).first()

    if not user:
        flash('Username not found')
        return redirect('generate_auth_token_and_send_it')

    notifier = EmailNotifier(sender=current_app.config['ADMIN_EMAIL'], as_html=True)
    tokenizer = TokenGenerator(secret_key=current_app.secret_key)

    send_user_reset_token(user, tokenizer, notifier)
    return redirect('index.homepage')

And our tests against the actual logic become really easy.

from collections import namedtuple

def send_user_reset_token(user, tokenizer, notifier):
    token = tokenizer.dumps({'op': 'reset-token', 'username': user.username})
    payload = 'Your reset token is {0}'.format(token)
    return notifier.notify(user, token)

DummyUser = namedtuple('DummyUser', ['username', 'email'])
DummyNotifier = namedtuple('DummyNotifier', ['notify'])
DummyTokenizer = namedtuple('DummyTokenizer', ['dumps', 'loads'])

def test_send_user_reset_token():
    user = DummyUser('fred', 'fred@fred.fred')
    notifier = DummyNotifier(lambda user, token: (user, token))
    tokenizer = DummyTokenizer(loads=lambda data: data, dumps=lambda data: data)

    result = send_user_reset_token(user, tokenizer, notifier)
    assert result == (user, {'op': 'reset-token', 'username': 'fred'})

This is a collaboration test not a unit test because our send_user_reset_token relies on the behavior of others, but it’s isolated completely from the outside world. We can test that it behaves correctly by simply handing it things we know work. If we wanted to make sure it talks correctly to these things, we could simply use a spying double and assert that each bit was called.

If it were more complicated, say it looped through some data set, we could pass it five different kinds of data sets to see how it behaves with each (those five are: empty, one, many, lots, and blows up; this is one of things Integrated Tests are a Scam taught me).

So let’s stop yanking out the complexity and simply shoving it someplace else and saying, “Look how clean it is! …no, no, no don’t pop open the decorators file!” Instead, let’s look the complexity in the eye and say, “I’m going to control you before you control me!”

Now, if you pardon me, I have a closet to organize.
